## Principle

Core Framework works best as a token and utility generator that supports a clear class architecture. For professional WordPress builds, default to semantic HTML plus BEM/custom classes powered by Core Framework variables. Use utilities as helpers, not as a replacement for component ownership.

## BEM Plus Variables

Use a block class for every reusable section or component:

```html
<section class="service-overview">
  <div class="service-overview__inner">
    <article class="service-card service-card--featured">
      <h3 class="service-card__title">Strategy</h3>
    </article>
  </div>
</section>
```

Write CSS with tokens:

```css
.service-card {
  display: grid;
  gap: var(--space-s);
  padding: var(--space-m);
  border-radius: var(--radius-m);
  background: var(--bg-card);
}

.service-card--featured {
  background: var(--primary);
  color: var(--text-on-primary);
}
```

If a token is missing, inspect the project before inventing one. Add a new semantic token only when it will be reused.

## Utility Classes

Use utilities when:

- The generated Core Framework output includes the class.
- The class expresses one clear, reusable behavior.
- The utility reduces noise without hiding component intent.

Avoid long utility-only class strings on complex components. They become hard to maintain in Bricks/Gutenberg and make design-system changes harder to audit.

Good hybrid pattern:

```html
<ul class="feature-grid">
  <li class="feature-card">...</li>
</ul>
```

```css
.feature-grid {
  display: grid;
  gap: var(--space-l);
  grid-template-columns: repeat(3, minmax(0, 1fr));
}
```

## Components and Modifiers

Use modifier classes for meaningful variants:

- `.button-row--center`
- `.pricing-card--highlight`
- `.media-section--reverse`
- `.testimonial-card--compact`

Use attributes when the variant comes from component props or dynamic data:

```css
.cta-card[data-tone="brand"] {
  background: var(--primary);
}
```

Prefer attributes for many mutually exclusive prop values. Prefer modifier classes for a few hand-authored variants.

## Naming Rules

- Use lowercase kebab-case.
- Use BEM syntax: `block`, `block__element`, `block--modifier`.
- Name for meaning, not appearance: `pricing-card`, not `blue-box`.
- Keep section blocks page-meaningful: `hero`, `service-overview`, `case-study-list`.
- Keep reusable components context-neutral: `icon-card`, not `home-icon-card`.
- Do not encode Core Framework token names into class names unless the class is genuinely a utility.

## Anti-Patterns

Avoid:

- Copying Automatic.css utility names into a Core Framework project.
- Styling only Bricks element IDs for reusable patterns.
- Hardcoding repeated colors and spacing inside page custom CSS.
- Naming classes after temporary content.
- Adding utility classes that are not generated by the active Core Framework project.
- Changing global Core Framework tokens to fix one section.
- Importing `.core` files with overwrite behavior for a narrow styling bug.

## Review Checklist

Before finishing a Core Framework class edit:

- The markup remains semantic and builder-editable.
- The class name describes the component or section.
- Core Framework variables are used for colors, spacing, typography, widths, and radii.
- Utility classes are verified in generated CSS.
- The edit does not alter unrelated global classes or tokens.
- Responsive behavior is explicit.
- Dark/light mode still works if the project supports it.
